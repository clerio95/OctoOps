# OctoOps — Python Remaster Specification

Use this document to implement OctoOps from scratch in Python. It supersedes `RUST_REWRITE_PROMPT.md`. The architecture philosophy is carried over; the implementation language, module loading model, and transport roles are redesigned. Start clean — do not port business logic from the Rust version or the Python prototype.

---

## Why Python, Not Rust

The compile-all-into-one-binary constraint of the Rust version created friction:

- Adding a module required a full recompile on a dev machine with the Rust toolchain.
- The deploy box could only run binaries — it could not build.
- The single-binary model was the wrong default for a system designed to grow by dropping in new modules.

Python removes the compile step entirely. Modules are discovered at startup by scanning the filesystem. Dropping a new module folder in and restarting is sufficient — no toolchain, no rebuild.

---

## What This System Is

A modular, async bot runtime that:

- Receives commands from **Telegram** (the control plane — operators and admins only).
- Routes commands to feature modules.
- Sends responses back to the originating Telegram chat.
- Mirrors selected outputs to configured **WhatsApp** chats via the Whatsmeow bridge.
- Runs scheduled jobs from modules.
- Enforces role-based authorization.

Core design goals:

- Core is small, domain-neutral, and stable. Business logic lives only in modules.
- Modules are isolated — they never import each other. Cross-module communication goes through an event bus.
- A failing module, job, or listener must not crash the runtime.
- Ships as a Python package on Windows, managed by Windows Task Scheduler, running 24/7.

## Transport Architecture

OctoOps has two active transports and one reserved slot:

| Transport | Role | Status |
|---|---|---|
| `TelegramTransport` | Control plane — operators send commands, receive responses | Active |
| `WhatsAppTransport` | Output transport — bot pushes messages to WA chats | Active |
| `MetaApiTransport` | Reserved | Future |

### Telegram = Control Plane

- Operators and admins interact with the bot exclusively through Telegram.
- WhatsApp users **cannot** initiate commands. WhatsApp is output-only.
- When a command executes, the response is sent back to the Telegram chat.
- Modules or the router may additionally mirror a response to one or more configured WhatsApp chats. The mirror routing rules are not specified here — this is deferred to a later design session. The architecture must leave room for it.

### WhatsApp = Output Transport

- The `WhatsAppTransport` can only send messages — it does not receive or route commands.
- It wraps the Whatsmeow Go bridge over local HTTP REST (see bridge section).
- The core `send()` abstraction is used for both transports; routing decides which transport(s) a response goes to.

---

## Architecture Layers

```
[Telegram Bot API]           [WhatsApp Network]
       ↓ ↑                          ↑
[TelegramTransport]      [WhatsAppTransport]
       ↓                        ↑
[Router]              ←→   [Response Router]    (decides which transports get the response)
       ↓
[Module Handler]
       ↓
[Event Bus]                  [Scheduler]
```

The `Transport` ABC defines `send(response)` and `run(router, registry)`. The response router (inside core) decides which active transports to deliver a response to. Initially: Telegram always gets the response; WhatsApp only gets it if the response is explicitly marked for mirroring (details deferred).

## Transport ABC

```python
from abc import ABC, abstractmethod

class Transport(ABC):
    @abstractmethod
    async def run(self, router: "Router", registry: "Registry") -> None:
        """Start receiving loop. Blocks until shutdown."""

    @abstractmethod
    async def send(self, response: "Response") -> None:
        """Send a response via this transport."""

    @property
    @abstractmethod
    def name(self) -> str:
        """Unique transport identifier, e.g. 'telegram', 'whatsapp'."""
```

Transports are instantiated in `build_transports(config)` based on `config.toml`. Adding a new transport means implementing this ABC and wiring it into `build_transports` — no other core changes.

## Telegram Transport

- Library: `python-telegram-bot` v21+ (async).
- Bot token from `config.toml → [telegram] → bot_token`.
- Allowed Telegram user IDs from `config.toml → [core] → allowed_user_ids`.
- Incoming messages are parsed: first word is the command (strip `/` prefix), rest are args.
- If the message does not match a registered command, return a help hint silently.
- Dispatch through the `Router`. Response is sent back to the originating chat.
- Mirror behavior: if `response.mirror_to_whatsapp` is set, the core response router also delivers it to the WhatsApp transport. Mirror routing details are deferred.

## WhatsApp Transport — Whatsmeow Bridge

Identical sidecar design to the Rust version. The Go bridge is reused unchanged.

### Sidecar Lifecycle

1. At startup, look for `whatsmeow-bridge.exe` (Windows) or `whatsmeow-bridge` (Linux/Mac) in the configured path.
2. Spawn it as a child process via `asyncio.create_subprocess_exec`.
3. Monitor it — if it exits unexpectedly, restart with exponential backoff (cap 60 s).
4. On OctoOps shutdown, send `POST /shutdown` to the bridge, wait up to 3 s, then `terminate()`.

### IPC Protocol (Local HTTP REST)

**Bridge exposes (Go side):**

| Method | Path | Description |
|---|---|---|
| `POST` | `/send` | Send a WA message. Body: `{ "chat_id": "...", "text": "..." }` |
| `GET` | `/health` | Liveness. Returns `{ "ok": true, "logged_in": true }` |
| `POST` | `/register-callback` | OctoOps registers its incoming-message webhook URL |
| `POST` | `/shutdown` | Graceful bridge stop |

**OctoOps exposes (Python/aiohttp side):**

| Method | Path | Description |
|---|---|---|
| `POST` | `/incoming` | Bridge calls this for each received WA message (reserved — not dispatched to router in current design) |

The `/incoming` endpoint exists for future use (e.g., if WhatsApp users are later permitted to trigger certain commands). For now it accepts and acknowledges messages without routing them.

## Module System

### Discovery

At startup, `plugin_loader.py` scans the `modules/` directory:

1. For each subdirectory containing a `plugin.json`, read the manifest.
2. If the module name is in `config.toml → [modules] → enabled`, proceed.
3. Import the module package (`importlib.import_module`).
4. Call its `load(ctx: ModuleContext) -> ModuleRegistration` function.
5. A broken import or failed `load()` logs the error and skips the module — runtime continues.

### plugin.json

```json
{
  "name": "my_module",
  "version": "1.0.0",
  "description": "One-line human description."
}
```

The `enabled` gate is in `config.toml`, not in `plugin.json`. `plugin.json` is metadata only.

### Module Contract

```python
from octoops.core.contracts import (
    ModuleRegistration, CommandDef, JobDef, ListenerDef, ConfigField,
    ConfigFieldKind
)
from octoops.core.registry import ModuleContext

def load(ctx: ModuleContext) -> ModuleRegistration:
    return ModuleRegistration(
        name="my_module",
        commands=[
            CommandDef(
                name="hello",
                description="Greet the user.",
                min_role=Role.Viewer,
                handler=handle_hello,
            )
        ],
        config_fields=[
            ConfigField(
                key="device_ip",
                label="Device IP",
                description="IP address of the target device.",
                required=True,
                default=None,
                kind=ConfigFieldKind.IpAddress,
            )
        ],
    )

async def handle_hello(request: Request, ctx: ModuleContext) -> Response:
    ip = ctx.config.require("device_ip")
    return Response(text=f"Hello from {ip}!", chat_id=request.chat_id)
```

### Core Contracts (contracts.py)

```python
from dataclasses import dataclass, field
from typing import Callable, Awaitable, Optional, Any
from enum import Enum

class ConfigFieldKind(Enum):
    Text = "text"
    Password = "password"
    FilePath = "filepath"
    IpAddress = "ipaddress"
    Integer = "integer"
    Boolean = "boolean"

@dataclass
class ConfigField:
    key: str
    label: str
    description: str
    required: bool
    default: Optional[str]
    kind: ConfigFieldKind

@dataclass
class CommandDef:
    name: str
    description: str
    min_role: "Role"
    handler: Callable[["Request", "ModuleContext"], Awaitable["Response"]]

@dataclass
class JobDef:
    name: str
    schedule: str           # cron expression
    handler: Callable[["ModuleContext"], Awaitable[None]]

@dataclass
class ListenerDef:
    event: str
    handler: Callable[["EventPayload", "ModuleContext"], Awaitable[None]]

@dataclass
class ModuleRegistration:
    name: str
    commands: list[CommandDef] = field(default_factory=list)
    listeners: list[ListenerDef] = field(default_factory=list)
    jobs: list[JobDef] = field(default_factory=list)
    config_fields: list[ConfigField] = field(default_factory=list)
    on_startup: Optional[Callable[["ModuleContext"], Awaitable[None]]] = None
    on_shutdown: Optional[Callable[["ModuleContext"], Awaitable[None]]] = None
```

## Role-Based Authorization

```python
from enum import IntEnum

class Role(IntEnum):
    Viewer = 1
    Operator = 2
    Admin = 3
```

- Check: `user_role >= command.min_role`.
- Fail closed — if a user's role cannot be determined, deny and log.
- Roles assigned by `config.toml → [core]` ID lists. A user ID present in multiple lists takes the highest role.
- The Telegram user ID is the `UserId`. WhatsApp JIDs are irrelevant for auth (WA is output-only).

## Request / Response (Internal Types)

```python
from dataclasses import dataclass, field
from typing import Optional
from enum import Enum

class TransportSource(Enum):
    Telegram = "telegram"
    WhatsApp = "whatsapp"

@dataclass
class Request:
    command: str
    args: list[str]
    raw_text: str
    user_id: str          # Telegram user ID (string)
    chat_id: str          # Telegram chat ID
    source: TransportSource

@dataclass
class Response:
    text: str
    chat_id: str
    reply_to: Optional[str] = None       # message ID to reply to
    mirror_to_whatsapp: bool = False     # if True, core also delivers to WhatsApp transport
    whatsapp_chat_ids: list[str] = field(default_factory=list)  # target WA chats (if mirroring)
```

The `mirror_to_whatsapp` / `whatsapp_chat_ids` fields are the extension point for the Telegram → WhatsApp relay feature (design deferred). Modules set these; core routes accordingly.

## Event Bus

```python
import asyncio
from typing import Any

EventPayload = Any   # typically a dict

class EventBus:
    async def publish(self, event: str, payload: EventPayload) -> None:
        """Fan-out to all listeners for this event. Each runs in its own task."""

    def subscribe(self, event: str, handler: Callable) -> None:
        """Register an async listener. Called during module load."""
```

- Each listener runs in its own `asyncio.create_task`.
- Exception in a listener is caught, logged, and does not affect others.
- Event names describe business meaning: `inventory.low`, `shift.started` — not transport events.

## Scheduler

- Library: `APScheduler` v4 (async-native).
- All jobs use the timezone from `config.toml → [core] → timezone`.
- Each job runs in its own task. A failing job is caught, logged, and the scheduler continues.

```python
class Scheduler:
    async def add_job(self, job: JobDef, ctx: ModuleContext) -> None: ...
    async def start(self) -> None: ...
    async def shutdown(self) -> None: ...
```

## Router

```python
class Router:
    def register(self, command_def: CommandDef, ctx: ModuleContext) -> None:
        """Fatal if command name already registered by another module."""

    async def dispatch(self, request: Request) -> Response:
        """
        1. Resolve user role from registry.
        2. Authorize — deny with a safe message if role insufficient.
        3. Call handler inside try/except — return fallback message on error.
        4. Log: received / completed / failed with latency.
        """
```

- Duplicate command registration raises `OctoOpsError` at startup (fatal).
- All commands are prefix-matched after stripping the `/` character.

## Registry

Centralized dependency container passed to every module context.

```python
class Registry:
    config: "AppConfig"
    event_bus: EventBus
    scheduler: Scheduler
    transports: dict[str, Transport]   # keyed by transport name
    start_time: datetime
```

Modules access the WhatsApp transport (for proactive sends, e.g., from scheduled jobs) via `registry.transports["whatsapp"].send(response)`.

## Startup Sequence (bootstrap.py)

1. Parse CLI args — `--setup` flag or `config.toml` absent → run wizard, then proceed.
2. Load and validate `config.toml` — fail fast on missing required keys, exit code 1.
3. Build `Registry`: config, event bus, scheduler, transport stubs.
4. Scan `modules/` directory; read `plugin.json` manifests; filter by `[modules] enabled`.
5. For each enabled module: `importlib.import_module`, call `load(ctx)`, collect `ModuleRegistration`.
   - Broken import or failed load → log error, skip module, continue.
6. Register commands into `Router` — duplicate = fatal, exit.
7. Register event listeners into `EventBus`.
8. Register jobs into `Scheduler`.
9. Run `on_startup` hooks (in registration order).
10. `await scheduler.start()`.
11. Build real transport instances (`TelegramTransport`, `WhatsAppTransport`).
12. Start WhatsApp adapter (spawn bridge, register callback, begin monitoring).
13. Start Telegram polling loop as an asyncio task.
14. Await shutdown signal (SIGINT / SIGTERM / Windows service stop via `win32api` on Windows).
15. Shutdown sequence: `on_shutdown` hooks → stop scheduler → graceful bridge shutdown → cancel tasks.

## Configuration Wizard (Textual TUI)

### Trigger

- Auto-opens when `config.toml` is absent.
- Re-opens on `--setup`.
- Completes → writes `config.toml` → bootstrap proceeds normally.

### Screens (in order)

1. **Welcome** — intro screen; if reconfiguring, warn that existing config will be overwritten.
2. **Telegram** — bot token (masked), admin chat ID (for notifications).
3. **WhatsApp Bridge** — bridge binary path (pre-filled `./whatsmeow-bridge.exe`), bridge port, callback port.
4. **Core Settings** — timezone (IANA string), allowed Telegram user IDs, operator IDs, admin IDs, default role, log file path.
5. **Module Selection** — checkbox list of all discovered modules in `modules/`. Checked = enabled.
6. **Module Config** — for each enabled module that declares `config_fields`, render each field dynamically. Required fields marked. Password fields masked. Optional fields show default.
7. **Task Scheduler** — yes/no: register in Windows Task Scheduler (SYSTEM, at startup). Runs `schtasks /create` if yes. Linux: this screen is skipped.
8. **Summary** — show full `config.toml` preview. Confirm to write.

The wizard contains **no module-specific knowledge**. All module fields come from `ConfigField` declarations in the module's `load()` function, collected during the wizard's pre-scan phase.

### QR / Pairing

After the wizard writes `config.toml`, if WhatsApp is enabled and the bridge is not yet authenticated, the wizard offers to run the pairing flow:

1. Restore the terminal (exit Textual).
2. Spawn the bridge — it prints its native QR code to stdout.
3. Poll `GET /health` until `logged_in: true` or timeout.
4. Report success or failure.

## Windows Task Scheduler

Target configuration (wizard generates via `schtasks /create` or writes an XML task file):

- **Trigger**: At system startup
- **Run as**: SYSTEM account
- **Action**: `python -m octoops` (full path to the Python interpreter in the venv)
- **Start in**: directory containing `config.toml`
- **On failure**: restart after 1 minute, up to 3 times

On non-Windows, `task_scheduler.py` is a no-op.

## Logging

- Library: `structlog` + standard `logging` as backend.
- Output: rotating file (path from `config.toml → [core] → log_file`, max size `log_max_bytes`) + stderr in development.
- Redact before logging: bot tokens, WA session credentials, full message bodies, raw secrets.
- DO log: command received/completed/failed (with latency ms), module loaded/skipped, job executed/failed, auth denied, bridge restarted, transport startup/shutdown.

Format: `timestamp | level | component | event | key=value ...`

---

## Error Handling

- `OctoOpsError` — base error class. Subclasses for config, auth, transport, module load errors.
- Handlers, jobs, and listeners are `async def` functions returning values, never raising to the runtime.
- Router wraps every handler call in `try/except Exception` — errors are logged, a safe fallback message is returned.
- Scheduler wraps every job call in `try/except Exception` — logs and continues.
- Event bus wraps every listener task in `try/except Exception` — logs and continues.
- Missing required config key at startup → `OctoOpsError`, logged, exit code 1.
- Missing optional config key at startup → warning only.

## Module Hard Rules

A module:

- **May**: define commands, subscribe to events, publish events, register jobs, declare config fields, use startup/shutdown hooks, call `registry.transports["whatsapp"].send()` for proactive WA messages.
- **Must NOT**: import another module.
- **Must NOT**: read `config.toml` directly — use `ctx.config.get()` / `ctx.config.require()`.
- **Must NOT**: create its own HTTP client, Telegram client, or database connection.
- **Must NOT**: register transport-level handlers directly.
- **Must NOT**: raise uncaught exceptions from handlers, jobs, or listeners.
- **Must NOT**: add dependencies not already in `requirements.txt`.

## What NOT to Port

Do not carry over from the Rust version or Python prototype:

- Any module business logic (deadlines, occurrences, healthcheck, vdroot, horustech, mix, gas station, etc.)
- Any domain-specific data formats
- Any Telegram-specific command implementations from the old prototype
- Any hardcoded group IDs or user IDs
- The `inventory` / compile-time registration pattern from the Rust version

The `modules/` directory starts with only the built-in `status` module: reports uptime, loaded modules list, and the requester's role.

## Design Principles (Non-Negotiable)

1. **Core Stability** — Core is small, domain-neutral, and stable. Business logic never touches core.
2. **Module Isolation** — Modules do not import each other. Cross-module communication goes through the event bus only.
3. **Error Boundaries** — A failing command, listener, or job must not crash the core or any other module.
4. **Centralized Dependencies** — All config, clients, and utilities flow through the registry. Modules never build their own.
5. **Transport Agnostic Core** — Core knows nothing about Telegram or WhatsApp. Transports are the only WA/TG-aware layer.
6. **Fail Closed** — When authorization is ambiguous, deny. When required config is missing, fail at startup, not at runtime.
7. **No Unhandled Exceptions in Handlers** — Handlers, jobs, and listeners catch their own errors and return gracefully.
8. **One Entry Point** — `python -m octoops`. The wizard, bot runtime, and bridge lifecycle are all managed from here.
9. **Modules Are Hot-Loadable** — Adding a module requires only dropping a folder into `modules/` and restarting. No recompile, no toolchain on the deploy machine.
